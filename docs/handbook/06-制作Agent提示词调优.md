# 第06章 制作 Agent 系统提示词调优：你能调的是流水线的骨头，不是肌肉

## 本章导览

到这一章，你已经知道怎么填好立项书、怎么把小说调成像样的剧本。现在我们走进 Toonflow 最重的一块机器：**制作 Agent**——它把一份剧本，串行地变成导演规划、衍生资产、分镜表、分镜面板，最后吐出分镜图。

很多人改提示词的姿势是错的：他们看到出图风格不对，跑去改 `production_execution_storyboard_gen.md`（那只是个"发起生成"的开关，根本不管画风）；他们嫌分镜太碎，跑去改决策层（决策层压根不拆分镜）。结果改了半天没动到正主，还把流水线的串行约束搞崩了。

要避免这种事，你得先建立一个核心认知——

**制作 Agent 的系统提示词（这 8 个 `production_*.md`）是"流程编排骨架"，不是"专业 know-how"。**

- **骨架**（本章 8 个文件）：管的是"谁先谁后、读什么数据、产出什么结构、按什么格式写、什么时候停"。它决定流水线**怎么跑**。
- **肌肉**（视觉/导演手册类技能，如 `art_character.md`、`director_storyboard.md`、`storyboard_prompt_techniques.md`）：管的是"画风长什么样、提示词怎么写漂亮、镜头语言怎么高级"。它决定产物**长什么样**。

二者在同一次调用里**叠加**：执行层先被注入骨架（流程＋格式＋红线），再按需激活肌肉（专业技法）。本章主攻骨架——它的调优收益高、踩坑也最狠，因为**改坏骨架会让整条流水线串行断裂、XML 输出破碎、监督层误判**，而改坏肌肉顶多是这一张图丑了。

本章按六阶段＋两层的真实文件，逐个讲清"可调旋钮"：它定义了什么规则、你想要"更细的分镜 / 更省图 / 更准的资产衍生 / 不同的转场风格"时该改哪一段、改动会引发什么连锁反应。文件路径全部以反引号给出，按需 Read 后再动手。

先看全景。制作 Agent 在 `src/agents/productionAgent/`，三层架构、六阶段串行：

```
[决策层] production_agent_decision.md          ← 唯一对接用户、调度六阶段、把控质量门
   │
   ├─ 阶段1 导演规划   directorPlanAgent      production_execution_director_plan.md
   ├─ 阶段2 衍生资产分析 deriveAssetsAgent      production_execution_derive_assets.md
   ├─ 阶段3 衍生资产生成 generateAssetsAgent    production_execution_generate_assets.md   (可选/异步)
   ├─ 阶段4 构建分镜表   storyboardTableAgent   production_execution_storyboard_table.md   ★唯一过监督质检
   ├─ 阶段5 分镜面板写入 storyboardPanelAgent   production_execution_storyboard_panel.md   (A/C 双模式)
   └─ 阶段6 分镜图生成   storyboardGenAgent     production_execution_storyboard_gen.md     (异步)

[监督层] production_agent_supervision.md        ← 只审分镜表，只提问题不改东西
```

记住三件事，本章后面所有判断都从这里推：

1. **执行层近乎无状态**：每个执行子 Agent 读一次工作区 → 产出 → 返回确认就终止；阶段间状态全靠落库（`o_agentWorkData`）和工作区数据（`get_flowData`）传递。所以"上下文记忆"不是靠 prompt 里塞历史，而是靠下一阶段去读上一阶段写好的数据。
2. **决策层不碰数据**：它不调 `get_flowData`、不操作资产、不做执行层的判断。它只读"执行层返回了什么结论"，据此调度下一步。
3. **A/C 模式由"多参（isRef）"决定**，且**模式由决策层指定、执行层不自行判断**——这是全章最容易改坏的地方，单列一节讲。

## 一、决策层：`production_agent_decision.md` —— 流水线的总调度

### 它定义了什么

这是整台机器的大脑。它规定了：六阶段的**调度工具映射**（阶段 N → `run_sub_agent_xxx`）、**串行约束**（六阶段必须按顺序）、**与用户的交互节奏**（哪些阶段要停下来等用户确认）、**质量门**（哪个阶段产出后自动派监督层）、以及一堆**全局红线**（缺资产不审核、视频生成不归我管等）。

它**不做**具体工作：不读工作区、不拆分镜、不写资产。它的全部动作就是"理解意图 → 选对调度工具 → 派 100 字以内的指令 → 接收执行层结论 → 决定下一步"。

### 可调旋钮一览

| 旋钮 | 在文件中的位置 | 控制什么 | 创作/制作意义 |
|---|---|---|---|
| 六阶段顺序与工具映射 | "制作流水线" + "调度与派发规范" 的工具表 | 谁先谁后、用哪个 `run_sub_agent_*` | 决定流水线拓扑，**几乎不该动** |
| 哪些阶段要"停下来等用户" | 各阶段"决策层行为"小节 | 交互节奏（阶段2/4/5 后是否暂停） | 多停＝可控但啰嗦；少停＝顺滑但失控 |
| 质量门位置 | "审核规则" + 阶段4 的"审核：需要" | 哪个阶段产出后自动派监督层 | 当前只有阶段4 过质检 |
| 缺资产不审核 | "全局约束" 第2条 | 基础资产缺失时不报问题、不建议新增 | 防止 AI 反复纠缠你"要不要补个资产" |
| 视频生成拦截 | "调度决策树" + "与用户交互规范" 第5条 | 用户说"生成视频"时拒绝并引导去视频面板 | 视频不在本流水线内 |
| 多参 → 写入模式映射 | 阶段5 "决策层行为" 的模型参数表 | `多参=是`→纯文本多参模式；`=否`→首位帧模式 | **最危险的旋钮，详见第九节** |
| 派发指令字数上限 | "调度与派发规范" | 派给执行/监督的指令正文 ≤100 字 | 执行层已有完整技能，指令只需点题 |

### 怎么调

**场景 A：你想让流水线"少打断我"，一路跑到底。**
默认在阶段2（衍生资产分析后）和阶段4（分镜表审核后）会停下来等你。如果你做的是批量、信任度高的项目，想减少确认：

- 找到阶段2 的"用户确认分支"表，可以把"分析结果需展示给用户确认是否进入图片生成"的硬约束放宽——但**注意 line 79 的约束**："不可自动进入阶段3"。这是有道理的：阶段3 是**花钱的异步出图**，自动进去等于自动烧 token/算力。建议保留这道闸。

**场景 B：你想增加一道质量门**（比如让导演规划也过监督层）。
当前"审核规则"写死"仅阶段4需要审核"。但要命的是——**监督层 `production_agent_supervision.md` 目前只实现了"分镜表审核"**（见下一节）。你在决策层加一句"阶段1也派监督"，监督层收到后会因为匹配不到审核对象而返回 `无法识别审核对象`。所以这是**跨文件联动改动**：要先在监督层补一套"导演规划审核"的维度表，决策层才能真正用上。只改决策层＝白改。

### Before / After

> **场景**：用户发来"把第3场的台词换一下"。

| | 内容 |
|---|---|
| ❌ Before（容易写出的坏调度） | 决策层直接自己读分镜表、自己改台词 |
| ✅ After（符合本文件铁律） | 决策层定位到"修改 X → 对应阶段4"，用 `run_sub_agent_storyboard_table` 派发**修复指令**（按"修复派发格式"模板，只带用户确认要改的项），自己不碰数据 |

### 常见错误

- **让决策层做执行层的判断**。文件 line 8 明文："决策层不做执行层的判断，执行层返回什么结论就基于该结论决策。" 你若在决策层 prompt 里加"如果分镜太碎就自己合并"，等于让大脑去干手的活，破坏分层。
- **改乱工具映射**。把阶段5 的工具误写成 `run_sub_agent_storyboard_gen`，会直接跳过面板写入去出图，而此时面板是空的——整条链断。
- **删掉"派发指令 ≤100 字"**。执行层技能是自带的，决策层把整份上下文塞进派发指令，既浪费 token 又可能与执行层自身技能打架。

## 二、监督层：`production_agent_supervision.md` —— 唯一的质检员（只管分镜表）

### 它定义了什么

监督层是制作侧唯一的质检。核心人设一句话：**只提问题和建议，不做任何修改决策，所有修改决定权属于用户。**

它定义了三块东西，全是你能调的：

1. **审核对象路由**（"审核任务识别"表）：目前只认"分镜表"一种对象。
2. **绝对红线 R1~R4**（"Skills（绝对红线）"）：违反即"严重"，一票否决。
3. **分镜表审核维度表**（"分镜表审核 → 审核维度"）：20 来项分级质检（严重/中等/轻微），每项带标准和红线关联。
4. **审核报告格式 + 评分标准**（A/B/C/D）。

### 可调旋钮一览

| 旋钮 | 位置 | 控制什么 |
|---|---|---|
| 红线 R1~R4 | "Skills（绝对红线）" | 哪些问题"违反即不可用"，一票判严重 |
| 审核维度表 | "分镜表审核 → 审核维度" | 每一项质检的标准与严重程度 |
| 评分阈值 | "评分标准" 表 | 多少严重/中等问题对应 A/B/C/D |
| 报告格式 | "审核报告格式" | 报告长什么样、是否输出"需要您决定"区块 |
| 缺资产不审核 | "通用审核原则" 第7条 + R1 多处 | 基础资产缺失时一律不报问题 |
| 审核对象路由 | "审核任务识别" 表 | 监督层能审哪些产物（目前仅分镜表） |

### 怎么调

**场景 A：你觉得监督层太松/太严。**
直接调"评分标准"表的阈值。比如默认 A 级允许 ≤2 个中等问题；你做的是高标准商单，想更严，可以改成 A 级要求 0 严重 0 中等。但别动红线分级逻辑——红线是"能不能用"的硬底线，不是"完不完美"。

**场景 B：你想新增一条质检维度**（例如"每场至少有一个特写镜"）。
在"审核维度"表加一行，写清楚 `审核项 / 严重程度 / 标准 / 红线`，再在"验证方法"区补一段怎么验。**注意**：维度表里的字段口径必须对齐**新分镜表结构**——文件 line 124 反复强调"勿再套用旧字段名 `associateAssetsIds`/`description`/`lines`/`sound`"。新结构是：场景信息在**场头**、资产引用在**片段级**两行、镜表无"朝向/空间关系/角色动作"独立列。你新增维度若按旧字段写，监督层会去找根本不存在的列，全表误判。

**场景 C：你想让监督层也能审"导演规划"或"分镜面板"。**
这是把监督层从"单对象"扩成"多对象"。要做三件事：①"审核任务识别"表加一行新标识词→新审核对象；②新增一节"XX审核"（数据准备 + 审核维度 + 验证方法）；③回到决策层把对应阶段标成"审核：需要"并派发。三处缺一不可。

### Before / After

> **场景**：分镜表里出现了 assets 中根本不存在的角色"何鸿燊"（基础资产缺失）。

| | 监督层行为 |
|---|---|
| ❌ Before（错误质检） | 报"缺少角色资产，建议新增何鸿燊资产" |
| ✅ After（符合 line 80/92 红线豁免） | **不报告**。基础资产是流程外输入，无任何阶段可新增，监督层一律不审"缺资产"。只在该资产**已存在**时才审它的引用/关联/衍生 |

### 常见错误

- **删掉"缺资产不审核"**。这会让监督层在每次审核都纠缠你"补资产"，而流水线根本没有"加基础资产"的阶段，陷入死循环。
- **红线和维度混改**。R1~R4 是"命中即严重、无视维度表严重程度列"的特殊通道（line 30）。你把一条普通维度误标成红线关联 R1，会让一个本该"轻微"的问题被一票判严重，评分被拉到 C/D，逼用户重做。
- **改报告格式时漏掉"审核通过的项不出现在报告中"**（精简规则）。一旦漏掉，报告会变成几十行流水账，淹没真正的问题。

## 三、阶段1 导演规划：`production_execution_director_plan.md` —— 只拆场、不拍镜

### 它定义了什么

导演规划是流水线第一道工序，人设是"50年经验的导演"，但**职责被刻意收窄到只做四件事**：①拆分场 ②逐场台词统计 ③逐场情绪分析 ④场间过渡设计。产物是一份**只给下游 AI 读、不给人读**的结构化 `<scriptPlan>`，含三节：分场汇总表、逐场注意事项、场间过渡表。

关键约束：**它不设计镜头**（景别/运镜/画面/拆镜全留给阶段4），**它不加载任何技法/skill**，**全程只允许两个动作**——读一次 `script` + 写一次 `<scriptPlan>`。

### 可调旋钮一览

| 旋钮 | 位置 | 控制什么 |
|---|---|---|
| 分场原则 | "分场原则（怎么切场）" | 场怎么切：地点变更/时间跳变/戏剧单元收束 |
| 情绪分析口径 | "情绪分析口径" | 情绪浓度 0~10、一句话基调、X→Y 推进 |
| 场间过渡设计 | "场间过渡设计" | 转场风格（动作衔接/空镜/淡入淡出叠化），及"唯一允许创作"的边界 |
| 该场注意事项 | "该场注意事项" | 下游要留意的点：情感砸点/一致性锚点/空间距离/环境音/易错提示 |
| 输出结构 | "输出结构" | `<scriptPlan>` 的字段与表格格式 |

### 怎么调

**场景 A：你想要"不同的转场风格"。**
这是本文件最适合创作者调的旋钮。改"场间过渡设计"那一段。默认给了三类（动作衔接、空镜、淡入淡出/叠化）。你做悬疑片想多用"声音先行的硬切"，可以在这里增补一类过渡，并在"场间过渡"输出表的示例行里给个范本。

**重点提醒**：文件 line 79 的设计哲学是"**先判断是否必要，不必要不增补**，不为凑齐场间数而硬造过场"。如果你想要更密集的转场（每场都来个空镜），就把这句改成"每个场间都设计过渡"——但代价是片子会被过场切碎、节奏拖沓。多数情况建议保留"按需"。

**场景 B：你觉得情绪分析太粗。**
改"情绪分析口径"。默认是 0~10 浓度 + 一句话基调 + 可选 X→Y。你想要更细的情绪曲线，可以要求每场拆成"开场情绪 / 转折点 / 收束情绪"三段。但记住下游（阶段4）只把它当**情绪依据**用，过度精细的情绪标注下游未必消费得了。

### Before / After

> **场景**：导演规划越权，开始替分镜设计镜头。

| | 产出 |
|---|---|
| ❌ Before | `Sc3：男主拔剑，给一个仰角特写，缓推到他的眼睛……`（这是镜头设计，越权了） |
| ✅ After | `Sc3：情感砸点：男主第一次动杀心的瞬间；空间距离：两人三步对峙` （只给"该被拍出什么"，不给"怎么拍"） |

### 常见错误

- **让它加载技法/skill**。红线第1条明令禁止。导演规划一旦激活画风技法，会开始写视觉细节，侵入阶段4 的职责，还拖慢这个本该轻量的阶段。
- **让它"创作"剧情**。除"场间过渡"这唯一例外，场次/台词/情绪/场内剧情只能忠实拆分剧本。你若放宽这条，导演规划会开始脑补剧情，下游分镜表跟着把脑补内容拍出来，偏离原著。
- **破坏 `<scriptPlan>` 一次性完整输出**（红线第9条）。这个标签是**前端流式写入的外层标签**，拆成多次 XML 输出会让前端解析错乱。

## 四、阶段2 衍生资产分析：`production_execution_derive_assets.md` —— 全章规则最硬的一块

### 它定义了什么

这个阶段决定"哪些资产需要做视觉状态变体（衍生），哪些不做"，并逐条调 `add_deriveAsset` 写入。它的**提取范围规则极其严格、几乎是写死的**：

| 资产类型 | 是否衍生 | 提取范围 |
|---|---|---|
| **角色** | 是 | **仅变身状态**：①服装 ②变身特效 ③变形 |
| **场景** | 是 | **仅时间变体**（日→夜、黄昏、清晨） |
| **道具** | **否** | **一律不提取任何变体** |

核心原则一句话：衍生是父资产**稳定、可复用、资产级**的视觉状态变体，**不是**为某个镜头临时拆的局部特写、不是瞬时表情。

### 可调旋钮一览

| 旋钮 | 位置 | 控制什么 |
|---|---|---|
| 提取范围表 | "提取规则 → 提取范围（按资产类型）" | 角色/场景/道具各自衍生什么 |
| "一律不需要衍生"清单 | "提取规则" 的规则列表 | 局部特写、瞬时表情等明确排除项 |
| 每资产衍生数量上限 | 规则"每个资产 1~5 个衍生，宁缺勿滥" | 衍生爆炸的闸门 |
| 稳定/可复用/资产级判据 | 规则"必须同时满足" | 防止把"剧本重点"误当"需要衍生" |
| name/desc/type 字段规则 | "`add_deriveAsset` 入参要求" | 衍生命名与描述格式 |

### 怎么调

**场景 A：你想要"更准的资产衍生"（少误判、少漏判）。**
最值钱的一段是"常见误判原因"：*"把'剧本重点描写'误当成'需要衍生资产'。判断标准不是它是否重要，而是它是否属于父资产稳定、可复用、整体级的视觉状态。"* 如果你的项目反复出现"AI 把一个紧张特写做成衍生资产"这类误判，强化这一段的措辞、多给几个反例，比改别处都管用。

**场景 B：你想让道具也能衍生**（比如"完好的剑 → 断剑"是关键剧情）。
当前道具被**硬性排除**。要打开，得改三处且彼此一致：①"提取范围"表把道具改成"是 + 损坏/激活态变体"；②`type` 字段说明放开 `tool`；③"强制约束"里"道具一律不衍生"那句删掉。**只改一处会自相矛盾**——比如你只改了范围表，但"强制约束"仍写"道具一律不衍生、超出范围不得写入"，执行层会以更靠后的硬约束为准，照样不衍生。

> **连锁影响提醒**：放开道具衍生后，阶段4 的监督层 R4（父子资产选择）会开始审"断剑剧情是否用了断剑衍生 ID"。也就是说，衍生范围一旦放宽，质检面也跟着扩大——这是好事，但你要意识到下游会变严。

### Before / After

> **场景**：剧本里有一句"她惊恐地瞪大双眼"。

| | 行为 |
|---|---|
| ❌ Before（误判） | 创建衍生资产"角色·惊恐脸" |
| ✅ After（正确） | **不衍生**。瞬时表情属"一律不需要衍生"清单，交给分镜描述/prompt 表达即可 |

> **场景**：剧本里女主"撕掉校服，变身成战甲形态"，且后续多场都是战甲。

| | 行为 |
|---|---|
| ✅ 正确 | 创建角色衍生（type=role），name 如"战甲形态"，desc 写"较校服态·全身银色机甲＋背后能量翼"。因为它稳定、可复用、改变整体识别外观 |

### 常见错误

- **只输出分析、不调 `add_deriveAsset`**。强制约束写得很重："仅输出分析文字视为未完成任务"、"调用次数必须与新增条数一致"。你若在调优时弱化了"必须实际调用工具"的措辞，模型很容易退化成"我分析完了"却没真写入，下游拿不到衍生资产。
- **依赖导演规划/预划清单**。这个阶段是**自主判定**的（line 32、74）——直接读剧本和资产描述自己判断，不读 `scriptPlan`。你若让它去读导演规划，反而引入耦合和误导。
- **放宽数量上限导致衍生爆炸**。"1~5 个、宁缺勿滥"是有意的。每个衍生都要在阶段3 出图（花钱花时间），上限一松，成本失控。

## 五、阶段3 衍生资产生成：`production_execution_generate_assets.md` —— 一个出图开关

### 它定义了什么

这是个极薄的阶段：读 `assets` → 收集需要出图的资产 id → 调 `generate_assets_images({ ids })`（**异步，发起即返回**）。它**不做任何创作判断**，画风/质量全由肌肉层（`art_*.md` 技能）和模型决定。

### 可调旋钮 + 怎么调

可调的几乎只有一条：**"仅对有衍生状态且尚未生成图片的资产发起生成"**（"约束"小节）。这是个**省图/省钱闸门**——它保证不重复出已生成的图。如果你想"强制重出全部衍生图"（比如换了画风技能要刷新），可以临时把这条约束放宽成"对指定 ids 强制重出"。但默认务必保留"尚未生成才出"，否则每次进阶段3 都会把整库重出一遍。

### 常见错误

- **跑到这个文件来调画风**。错门。衍生图的视觉风格在 `art_character_derivative.md` / `art_scene_derivative.md`（肌肉层）和图像模型里，这个阶段只负责"按下生成键"。
- **以为它是同步的**。它是异步发起即返回。你若在调优里加"等待生成完成再返回"，会和异步机制冲突——决策层的设计就是"派发后告知用户等待"。

## 六、阶段4 构建分镜表：`production_execution_storyboard_table.md` —— 把剧本拆成镜头的主战场

### 它定义了什么

这是创作者最常想调、也是**唯一过监督质检**的阶段。它把剧本拆成结构化分镜表，写进 `<storyboardTable>`。它定义了一整套**核心信条与铁律**（12 条）＋**片段间过渡专项规则**（4 类）＋**严格六步线性流程**＋**输出格式**＋**红线**。

铁律有明确优先级（line 14）：**台词零删改 > 出场人物完整 > 只描述动作状态 > 长台词/长VO拆镜规则**。冲突时按此让步。

### 可调旋钮一览

| 旋钮 | 位置 | 控制什么 | 你想要…就调它 |
|---|---|---|---|
| 片段时长上限 | 铁律2 "每个片段 ≤15 秒" | 单片段最长时长 | 更短的节奏 |
| 长台词拆镜阈值 | 铁律3 ">20 字必须拆镜" | 长台词何时强制拆成多镜 | **更细的分镜** |
| 台词时间分配 | 铁律5 "4字/秒" | 台词时长换算语速 | 语速快慢 |
| 画面描述内容边界 | 铁律8/9 | 只写动作/姿态/表情/状态变化，不写外观 | 防外观混入提示词 |
| 声音两类规则 | 铁律10 | 只写环境音+音效，禁 BGM | 音效策略 |
| VO 音画同步 | 铁律11 | 旁白/面板文字怎么落地 | 系统流/面板流题材 |
| 片段间过渡专项 | "专项规则" 4 类 | 消灭跳跃感的桥梁元素 | 不同连贯风格 |
| 画面描述字数 | 输出格式（监督层 line 161/284 提到 15~50 字上限） | 单镜描述详略 | 更细/更粗的描述 |

### 怎么调（重点：你想要"更细的分镜"）

这是创作者问得最多的需求。**正确的旋钮在这里，不在决策层。** 几个动作：

1. **降低长台词拆镜阈值**：把铁律3 的">20 字"改成">12 字"，台词会被切成更多连续镜头，单镜信息密度下降、镜头数上升。
2. **收紧片段时长**：把铁律2 的"≤15 秒"改成"≤10 秒"，强制更频繁地切片段。
3. **强化"相邻镜景别视角错开"**（核心信条1）：让相邻镜更频繁换景别，观感更"碎"更动感。

**但务必同步看监督层的对应维度**——这是双向锁：

- 监督层有"长台词拆镜"（line 157）、"连贯优先/拆分粒度"（line 161）、"片段时长合理 ≤15s"（line 156）等维度。你在阶段4 把阈值改了，**监督层还按旧阈值审**，就会出现"分镜表照新规则拆了，监督层却报'拆太碎/片段超时'"的打架。改分镜表的量化阈值（字数、秒数），**记得回监督层同步同一个数**。

### 怎么调（你想要"更省"或"不同的连贯风格"）

- **更连贯、少碎镜**：强化"连贯优先"红线（红线5）和"专项规则"，提高合并力度。反过来想更碎就弱化它。注意监督层"过度拆分的信号 / 过度合并的信号"（line 282-290）也要同步。
- **换连贯风格**：改"专项规则"的四类桥梁（动作的桥梁/情绪的接力/空间视线链接/台词动作黏合）。比如做动作片想强调"动作不冻结"，就重点扩写"动作的桥梁"那段的正反例。

### Before / After

> **场景**：剧本一句长台词"我等了你整整十年，从青丝等到白发，你以为我还会回头吗"（28 字）。

| | 分镜表产出 |
|---|---|
| ❌ Before（违反铁律3） | 单镜固定机位、一口气念完 28 字、画面 7 秒不动 |
| ✅ After | 按语义停顿拆 3 个连续镜：镜1 近景念前半句（含字幕台词）、镜2 切听者反应镜（声音继续）、镜3 特写说话人眼眶——每镜换景别，填满时长不冷场 |

### 常见错误

- **改台词忠实度铁律**。台词零删改是**第一优先级铁律 + 红线R2**。你若为了"节奏"放宽成"可适当精简台词"，监督层 R2 会一票判严重，且偏离原著——这是产品红线，别碰。
- **在画面描述里写光影/色调/配乐**。红线7 明令禁止，监督层"禁光影色调""音效禁配乐"会命中判严重。原因是肌肉层设计：光影由场景图自动承担，文字描述会和场景图原生光影**打架**。
- **破坏 `<storyboardTable>` 是纯 markdown 的约束**（输出格式 line 119）："标签之间只放纯 markdown，禁止嵌套任何其他 XML 标签"。你若在里面塞 `<scene>` 之类标签，前端流式解析直接崩。
- **让它操作资产**。红线8："严禁创建/修改/删除/生成任何资产。" 它只能引用 assets 已有资产，缺的只在画面里体现、不编造 ID。

## 七、阶段5 分镜面板写入：`production_execution_storyboard_panel.md` —— 全章最危险的双模式路由

> 这是整章你最该小心的文件。它有 **A（纯文本多参）/ C（首位帧）两种模式**，模式由**决策层根据"多参（isRef）"指定、执行层不自行判断**。改错这里，要么白出一堆图浪费算力，要么该出的图一张没出。下一节专门讲 A/C 与多参的关系，这里先讲文件本身。

### 它定义了什么

把分镜表落成"分镜面板"（逐条调 `add_flowData_storyboard`，**不再输出 `<storyboardItem>` XML**）。第一步必做**路由**：识别决策层指令里的模式关键词，进入对应流程，**流程内零条件分支**。

| 派发模式 | 进入流程 | track 规则 | prompt | shouldGenerateImage | 写入单位 |
|---|---|---|---|---|---|
| **纯文本多参模式** | 流程 A | 按"组"顺序累加 | `null` | `"false"` | 分镜表的**组** |
| **首位帧模式** | 流程 C | 每行独立一组、递增 | 完整生成 | `"true"` | 分镜表的**数据行** |

`add_flowData_storyboard` 六个参数：`videoDesc`（必填）、`prompt`、`track`、`duration`、`associateAssetsIds`、`shouldGenerateImage`。

### 两个模式的本质区别

- **流程 A（纯文本多参）**：不加载技法、**不生成 prompt、不出分镜图**。以"组"为单位写入，每组拼一段 `videoDesc`（含"承接上镜段"过渡句 + 该组分镜行原文照搬），`shouldGenerateImage="false"`。这是给**支持多图参考的视频模型**直接用文字驱动的——不需要首帧图，省掉整个出图环节。
- **流程 C（首位帧）**：**激活肌肉层技法**（`storyboard_prompt_techniques` + 风格专属 `director_storyboard`），逐行生成 prompt、做人物位置/朝向预分析、`@图N` 标注、六项忠实性校验，`shouldGenerateImage="true"`。这是给**需要首帧图引导**的视频模型用的。

### 可调旋钮一览

| 旋钮 | 位置 | 控制什么 |
|---|---|---|
| 路由表（模式关键词→流程） | "路由（第一步必做）" | 哪个关键词进哪个流程 |
| 流程A 承接上镜段规则 | 流程A 第2步 | 组间过渡句怎么推导、覆盖哪三要素 |
| 流程A 原文搬运边界 | 流程A 第2步末 | 除承接句外不得改写原文 |
| 流程C 人物位置/朝向预分析 | 流程C 第2步 | 连贯性基准表怎么建 |
| 流程C `@图N` 标注规则 | 流程C 第4步 | 参考图与画面的绑定 |
| 流程C 六项忠实性校验 | 流程C 第6步 | prompt 必须对齐分镜表的 6 个维度 |
| 全模式共享硬约束 | 文末 | 数量一致性、时长一致性、光影/音乐排除 |

### 怎么调

**场景 A：流程 C 出来的图老是人物站位/朝向乱跳。**
强化流程C 第2步"人物空间位置与朝向预分析"和第6步校验项6（朝向一致）。这是连贯性的命门——基准表一旦没锁住，同一场里角色一会儿左一会儿右。

**场景 B：流程 A 的组间过渡太生硬/太啰嗦。**
调流程A 第2步"承接上镜段"的措辞。注意它的精妙设计：承接的是上组**末帧的静态定态**（位置、姿态、朝向），**不是续接进行中的动作弧线**（因为分组已保证一个连贯动作不跨组）。你若改成"承接上一组的动作过程"，会和分组逻辑冲突，产出错乱的过渡句。

### 常见错误（这里翻车代价最大）

- **改 `shouldGenerateImage` 取值**。它是**字符串枚举** `"true"`/`"false"`，不是布尔。你若手滑写成布尔 `true`，下游判断可能全部失效。
- **打破"数量一致性 / 时长一致性"硬约束**（文末）。A 模式调用次数 = 分镜表组数；C 模式 = 数据行数。你若让 A 模式也逐行写，组的语义就废了。
- **让 A 模式去激活技法/生成 prompt**。A 模式明令"不加载任何提示词技法"。你若给它加技法，它会开始生成 prompt 还把 `shouldGenerateImage` 搞成 true，等于把 A 拍成了四不像。
- **在 `videoDesc`/`prompt` 里写光影/色调/音乐**（共享硬约束）。理由同阶段4：和场景图原生光影冲突。

## 八、阶段6 分镜图生成：`production_execution_storyboard_gen.md` —— 又一个出图开关

### 它定义了什么

和阶段3 一样薄：读 `storyboard` → 提取**真实分镜 ID** → 调 `generate_storyboard_images({ ids })`（异步）。约束三条：前置是面板已写完、图要和分镜描述匹配、**只用真实 ID 禁止编造**。

### 怎么调 + 常见错误

- 它几乎没有创作旋钮。唯一值得守的是"仅使用真实分镜 ID、禁止编造或复用无效 ID"——这条务必保留，否则会对不存在的 ID 发起生成、报错或生成脏数据。
- **常见错误**：跑来这里调画风/调提示词。又错门。提示词在阶段5（流程C）就定好了，这一步只是"按生成键"。
- **注意它和阶段5 的耦合**：只有流程C（首位帧、`shouldGenerateImage="true"`）写入的面板才需要也才会有意义地走阶段6。如果阶段5 走的是流程A（纯文本多参、不出图），那本就**不该**进阶段6——这点决策层已处理（A 模式完成后是引导去视频工作台，而非询问出图）。

## 九、专题：A/C 模式 × 多参（isRef）—— 改错这里会怎样

这是全章最值得单独拎出来讲的耦合，因为它横跨决策层、阶段5、阶段6 三个文件，且**链路是隐式的**。

### 完整链路

```
项目设置里的"视频参考模式 mode"（即 isRef / 多参）
        │
        ▼
[决策层 阶段5]  多参=是 → 派"纯文本多参模式"   多参=否 → 派"首位帧模式"
        │
        ▼
[阶段5 路由]   纯文本多参 → 流程A（不出图, shouldGenerateImage=false, 按组写）
              首位帧     → 流程C（出图,   shouldGenerateImage=true,  逐行写, 激活技法)
        │
        ▼
[阶段6]        流程A 不进阶段6（去视频工作台）   流程C 进阶段6 出分镜图
```

### 为什么这样设计

- **多参（isRef=是）**：视频模型支持多图参考，可以直接吃文字 + 资产参考图生成视频，**不需要逐镜首帧图**。所以走流程A：省掉 prompt 生成和出图，最省、最快，直接去视频工作台。
- **非多参（isRef=否）**：视频模型需要一张"首帧图"作为每个镜头的起点。所以走流程C：逐镜生成 prompt → 出首帧图（阶段6）→ 拿去生成视频。

### 改错会怎样（务必背下来）

| 你改错的地方 | 后果 |
|---|---|
| 决策层把多参映射反了（多参=是→首位帧） | 多参项目白白逐镜出一堆首帧图，**浪费大量算力和时间**，且和视频模型期望的"文字多参"不符 |
| 决策层派发时漏带模式关键词 | 阶段5 路由无法识别模式，流程紊乱（执行层被明令"不自行判断模式"） |
| 阶段5 把流程A 的 `shouldGenerateImage` 改成 `"true"` | A 模式本不出图，却标了要出图，阶段6 会对没有 prompt 的面板发起出图，**出一堆空/错图** |
| 阶段5 让流程C 按"组"写而非"行" | 首帧图与镜头错位，一个 track 里塞多镜，出图与分镜对不上 |
| 阶段6 不加判断对所有面板出图 | 多参（流程A）项目也被拉去出图，**违背省图设计** |

> **一句话忠告**：多参/A/C 这条链是"模型能力 → 流程分叉"的硬绑定，不是创作偏好。**除非你清楚视频模型到底吃不吃多图参考，否则别动这条映射。** 想省图，正道是确认你的视频模型支持多参、然后在项目设置里把 mode 设成多参——而不是去改 prompt 文件。

## 十、骨架 × 肌肉：两类技能如何叠加进同一次调用

前面反复提到"肌肉层"，这里说清它和本章骨架的分工与叠加方式。

| | 骨架（本章 8 个 `production_*.md`） | 肌肉（视觉/导演手册技能） |
|---|---|---|
| 管什么 | 流程、顺序、读什么、产出什么结构、什么格式、什么时候停 | 画风、镜头语言、提示词写法、解析映射、景别词库、连贯性规则 |
| 典型文件 | `production_execution_storyboard_table.md` | `storyboard_prompt_techniques.md`、风格专属 `director_storyboard.md`、`art_character.md`、`art_scene*.md` |
| 何时注入 | 执行子 Agent 一启动就带着（它的"身份"） | **按需激活**：仅在需要专业 know-how 的阶段才加载 |
| 改坏的代价 | 流水线断、格式崩、监督误判（**全局性**） | 这一张图/这一段提示词丑（**局部性**） |

**叠加发生在哪一次调用？** 看流程里"是否激活技法"的明文：

- **不激活肌肉的阶段**：导演规划（`director_plan` 红线第1条明令不加载技法）、衍生分析、衍生出图、分镜表（红线第1条"未激活任何技法"）、分镜图生成、以及**分镜面板的流程A**（"本模式不加载任何提示词技法"）。这些阶段的全部规则**以骨架 prompt 为准**。
- **激活肌肉的阶段**：**分镜面板的流程C**——第1步明文"激活技法 `storyboard_prompt_techniques` + 风格专属 `director_storyboard`，冲突时以风格专属技法为准"。此时一次调用里同时叠了：骨架（流程C 的步骤＋格式＋校验）＋肌肉（怎么把分镜表字段映射成漂亮 prompt、怎么标 `@图N`、景别词库等）。

**给调优者的判断口诀**：

- "我想改**流程/格式/什么时候出图/分镜拆多细的规则**" → 改本章**骨架**文件。
- "我想改**画风长相 / prompt 写得高不高级 / 镜头语言**" → 改**肌肉**文件（多在阶段5 流程C 生效）。
- 想改的东西在"不激活技法"的阶段（如分镜表的镜头**拆分规则**）→ 一定在骨架里，去肌肉文件里翻是白费。

## 十二、调这些骨架级提示词的风险提示

骨架提示词是"流水线的骨头"，断一根整条腿就废。动手前把这几条钉在脑子里：

1. **串行约束不能破**。六阶段顺序、前置条件、调度工具映射，是流水线能跑通的前提。改决策层调度逻辑＝在做骨折手术，非必要不碰。

2. **XML 输出必须一次性完整、且边界纯净**。
   - `<scriptPlan>`、`<storyboardTable>` 都是**前端流式写入的外层标签**，红线明令"一次性完整输出、禁止拆成多次 XML"。
   - `<storyboardTable>` 内部"只放纯 markdown，禁止嵌套其他 XML 标签"。
   - 你在调优时若加了"分批输出""可以嵌套结构标签"之类的话，前端解析会直接崩。

3. **格式约束 = 下游能不能读懂的契约**。分镜表的新结构（场头/片段级资产/镜表列）、`add_flowData_storyboard` 的六参数及其类型（尤其 `shouldGenerateImage` 是**字符串枚举**）、衍生资产的 `name/desc/type` 格式——这些是阶段之间的"接口"。改一边不改另一边，下游就读错。**改字段格式是跨文件改动**，决策层↔监督层↔执行层要一起对齐。

4. **量化阈值要双向同步**。片段 ≤15s、长台词 >20 字拆镜、4字/秒、画面描述 15~50 字——分镜表（生产侧）和监督层（质检侧）各持一份。**只改一侧 = 自己造一个"生产出来就被判不合格"的死循环。**

5. **A/C/多参映射是模型能力的硬绑定，不是创作偏好**。见第九节。改它之前先搞清你的视频模型吃不吃多图参考。

6. **别让执行层加载它不该加载的技法**。"不激活技法"是多个阶段的红线（导演规划、分镜表、面板流程A）。违反会让轻量阶段变重、职责越界、产出风格漂移。

7. **缺资产豁免不能删**。决策层和监督层都写死"基础资产是流程外输入、无任何阶段可新增、缺了不审"。删掉这条，AI 会陷入"建议你补资产→没有补资产的阶段→再建议"的死循环。

8. **改完用 `yarn lint` 之外的方式验证**——这些是 markdown 提示词，不进 TypeScript 类型检查。验证方式是**跑一遍真实流水线**：建个小项目，从阶段1 跑到阶段6，观察每阶段产出格式是否正常、监督层会不会误判、A/C 模式出图行为是否符合预期。提示词改动没有编译器替你兜底，只有端到端跑一遍才算数。

> **最后的心法**：骨架提示词改动的"半径"远大于它的字数。一句话改动可能横跨三个文件、影响整条流水线。动手前先用第十一节的对照表确认"主改文件 + 需同步项"，改完务必端到端跑一遍。**慢就是快。**

## 本章调优速查表

| 主题 | 关键结论 |
|---|---|
| **骨架 vs 肌肉** | 8 个 `production_*.md` = 流程编排骨架；`art_*` / `director_storyboard` / `storyboard_prompt_techniques` = 专业肌肉。骨架管"怎么跑"，肌肉管"长什么样" |
| **决策层** | `production_agent_decision.md`：只调度不干活；六阶段串行；仅阶段4 过质检；多参→模式映射在此（高危） |
| **监督层** | `production_agent_supervision.md`：只审分镜表、只提问题不改；R1~R4 红线一票判严重；缺资产不审；评分阈值/维度可调 |
| **阶段1 导演规划** | 只拆场+情绪+台词统计+转场；不设计镜头、不加载技法；`<scriptPlan>` 一次性输出。想改转场风格→改"场间过渡设计" |
| **阶段2 衍生分析** | 角色仅变身/场景仅时段/道具不衍生；自主判定不读规划；必须实际调 `add_deriveAsset`；1~5个宁缺勿滥。开禁道具需三处同步改 |
| **阶段3 衍生出图** | 薄开关，异步；只对"有衍生且未出图"的资产生成。画风不在这 |
| **阶段4 分镜表** | 唯一过质检；铁律优先级：台词零删改>全员在场>只写动作>拆镜。**想要更细分镜→改铁律2/3 阈值，并同步监督层** |
| **阶段5 分镜面板** | 双模式路由，模式由决策层定；A=不出图按组写`false`、C=出图逐行写`true`+激活技法。**数量/时长一致性是硬约束** |
| **阶段6 分镜图** | 薄开关，异步；只用真实分镜 ID；仅流程C 面板进这步 |
| **A/C×多参** | 多参=是→流程A 省图；多参=否→流程C 出首帧图。改映射前先确认视频模型是否支持多图参考 |
| **最大的坑** | ①破坏串行 ②XML 拆分/嵌套 ③字段格式只改一侧 ④量化阈值生产/质检不同步 ⑤多参映射改反白烧算力 |
| **验证方式** | 提示词不进 `yarn lint`；唯一靠谱的验证是建小项目端到端跑一遍六阶段 |
