# 第08章 视频生成、配音与成片：后期总监的工作台

## 本章导览

到了这一章，剧本定了、资产出了、分镜也排好了。接下来是把"一张张静态分镜图"变成"会动、有声、能连成片"的活儿——这是后期总监真正下场的环节。

工作台（前端的"视频生成"页）背后是一组路由，全在 `src/routes/production/workbench/`。它们干三件事：

1. **生成视频提示词**（把分镜信息翻译成 AI 视频模型吃得懂的运动/镜头描述）；
2. **生成视频**（把提示词 + 参考图/首尾帧/音频喂给视频模型，产出 mp4 候选）；
3. **组织成片**（轨道制：每个分镜一条轨道，可生成多个候选挑一条，按顺序连成片）。

本章我会逐个拆解：

- **视频模型与 `mode` 的选择**——这是整章最关键的决策点。选错 `mode`，提示词模板会跟着错、参考图会被白白浪费、运动效果完全不对。
- **视频提示词机制**——模型自适应模板怎么选、怎么手动改、时长/分辨率/画幅怎么取舍。
- **配音与音频**——角色音色绑定、`audioReference` 用法、TTS 现状、对口型的注意点。
- **工作台协作**——一镜多候选选片、轨道增删、批量并发与失败重试、状态查询。
- **成片合成与导出现状**——以及一个老后期会主动提醒你的坑清单。

先记住一张总图（字段来自 `o_project`、`o_videoTrack`、`o_video` 表，定义在 `src/lib/initDB.ts`）：

| 概念 | 落在哪张表 | 一句话 |
|---|---|---|
| 视频模型 | `o_project.videoModel`（`vendorId:modelName`） | 用谁来出片 |
| 参考模式 | `o_project.mode`（字符串或 JSON 数组） | 决定参考图怎么用、提示词用哪套模板 |
| 画幅 | `o_project.videoRatio`（`16:9`/`9:16`） | 出片比例 |
| 一条镜头 | `o_videoTrack`（一条 = 一个分镜段） | 轨道，带 `prompt`/`duration`/`state` |
| 一个候选片 | `o_video`（多条挂在一条 track 下） | 同一镜头的不同生成结果 |
| 选定的片 | `o_videoTrack.videoId` | 这条轨道最终用哪个候选 |
| 角色音色 | `o_assetsRole2Audio`（角色资产 ↔ 音频资产） | 谁用哪条嗓子 |

> **关键认知**：工作台的一切都围着"轨道（track）"转。一条轨道 = 一个分镜段 = 成片里的一个镜头。轨道上可以反复生成多个 `o_video` 候选，最后用 `selectVideo` 钦定一条。理解了这个结构，后面所有操作都顺了。

## 一、视频模型与 `mode`：全章最重要的决策

### 1.1 `mode` 到底是什么

`o_project.mode` 是一个文本字段（`src/lib/initDB.ts` 中 `table.text("mode")`），但它有两种形态：

- **单值字符串**：如 `"singleImage"`、`"startEndRequired"`、`"text"`；
- **JSON 数组字符串**：如 `'["imageReference:9","audioReference:3"]'`，解析后是多参模式。

`getGenerateData.ts` 里有一段判定逻辑（约 46–51 行）：先 `JSON.parse(mode)`，**能解析成数组就是"多参模式"（`isRef = true`），否则按单值模式走**。这个 `isRef` 直接决定工作台要不要给你拉出资产参考图、要不要按 `audioReference:N` 的数量截取音频。

代码里 `VideoMode` 的完整定义在 `src/utils/ai.ts`（274–280 行）：

```ts
type VideoMode =
  | "singleImage"        // 单图参考
  | "startEndRequired"   // 首尾帧（两张都得有）
  | "endFrameOptional"   // 首尾帧（尾帧可选）
  | "startFrameOptional" // 首尾帧（首帧可选）
  | "text"               // 纯文本
  | (`videoReference:${number}` | `imageReference:${number}` | `audioReference:${number}`)[]; // 多参
```

### 1.2 每种 mode 详解

| mode | 控制什么 | 对参考素材的要求 | 适用场景 | 选错会怎样 |
|---|---|---|---|---|
| `singleImage` | 用一张图作首帧，AI 据此生成运动 | 必须恰好 1 张参考图（首帧） | 大多数"图生视频"分镜：人物说话、轻微运镜、表情动作 | 给了多张图也只用第一张，其余白传 |
| `startEndRequired` | 首帧 + 尾帧都必给，AI 补中间过渡 | **两张图都必须有**（起始画面 + 结束画面） | 明确的转场/变身/位移：从 A 状态过渡到 B 状态 | 只给一张会报错或退化；提示词模板按"首尾帧"生成，单图模型吃不下 |
| `endFrameOptional` | 首帧必给，尾帧可选 | 至少首帧 | 想要"有目标落点但不强制"的运动 | 当成 singleImage 用浪费了模板能力 |
| `startFrameOptional` | 尾帧必给，首帧可选 | 至少尾帧 | 倒推式：知道要停在哪个画面 | 同上 |
| `text` | 纯文生视频，无参考图 | 无图 | 没有分镜图、纯靠文字描述的镜头（如 Seedance 2.0 多参文本流） | 你精心做的资产图全用不上，人物会"跑形" |
| `imageReference:N` | 多图参考，最多 N 张 | 1~N 张图（角色图 + 场景图 + 道具图组合） | 多主体同框：两个角色对戏、角色 + 场景 + 道具组合 | 给超过 N 张会被 `getGenerateData.ts` 截断；给少了浪费一致性 |
| `videoReference:N` | 视频参考（首视频续写/风格参考） | 1~N 段视频 | 续拍、风格迁移、动作参考 | 用图当视频参考会类型不匹配 |
| `audioReference:K` | 音频参考，最多 K 条 | 绑定的角色音色音频 | 需要把台词音色/对口型喂给模型的镜头 | 不配 `audioReference` 时音频根本进不去 referenceList（见第三节） |

> 真实可选值参考 `src/lib/vendor.json` 里各供应商模板的 `VideoModel.mode` 定义。例如 `volcengineSd2.ts` 的某些视频模型声明了 `["singleImage","startEndRequired","endFrameOptional","startFrameOptional","text","imageReference:9","videoReference:3","audioReference:3"]`——也就是说一个模型可以同时支持多种 mode，项目里选哪个由 `o_project.mode` 锁定。

### 1.3 多参模式是数组，可以组合

多参模式最强也最容易踩坑的地方是：**它是数组，可以同时声明图参考和音频参考**。例如：

```json
["imageReference:9", "audioReference:3"]
```

含义是：这条镜头最多吃 9 张参考图 + 3 条音频。`getGenerateData.ts`（86–92 行）会解析出 `audioReferenceCount = 3`，并在拼装 track 媒体时按这个数量截取 audio 类型资产（184–191 行）。也就是说：**`audioReference:K` 里的 K 不只是上限，它还驱动工作台"放几条音频进参考列表"。** 写成 `audioReference:0` 或干脆不写，音频就进不去。

### 1.4 "想要的运动效果 → 选哪种 mode/模型"建议表

这张表是后期最常翻的：

| 你想要的效果 | 推荐 mode | 备注 |
|---|---|---|
| 人物站着说话、轻微表情/呼吸 | `singleImage` | 最稳，分镜图当首帧即可 |
| 镜头推/拉/摇 + 人物小动作 | `singleImage` + 提示词写明运镜 | 运镜靠 prompt，不靠 mode |
| 明确的场景转换/变身/开门进屋 | `startEndRequired` | 首帧 = 转换前，尾帧 = 转换后 |
| 角色 A 和角色 B 对戏同框 | `imageReference:N`（N≥2） | 把两个角色图都传进去保一致性 |
| 角色 + 特定场景 + 关键道具 | `imageReference:N` | 多主体组合，N 给够 |
| 带台词对口型 | `imageReference:N` + `audioReference:K` 组合 + `audio: true` | 见第三节，注意供应商是否真支持对口型 |
| 纯文字描述、无分镜图（如 Seedance 2.0） | `text` 或多参文本流 | 资产图作唯一参考素材，人物一致性靠 prompt |
| 续接上一段视频 | `videoReference:N` | 把前段视频当参考 |

**头号原则**：`mode` 必须和你选的 `videoModel` 的能力交集匹配。在创建项目（第01章）时就该把 `videoModel` 和 `mode` 一起定好——选了只支持首尾帧的模型却把 `mode` 写成 `imageReference:9`，到生成那一刻才发现模型不认，整批白跑。

## 二、视频提示词：模型自适应模板机制

### 2.1 提示词怎么生成的（流程）

视频提示词不是你手写的，是 AI 根据分镜信息生成的。入口是 `generateVideoPrompt.ts`（单条）和 `batchGeneratePrompt.ts`（批量，带并发）。流程一致：

1. 查分镜（`o_storyboard`：`videoDesc`/`prompt`/`duration` 等）和关联资产（`o_assets2Storyboard` → `o_assets`）；
2. 拆出 `assets`（角色/场景/道具/音频）和 `storyboard`（分镜列表）；
3. **选用哪套提示词模板**（自适应，见下）；
4. 拼一段 `content`（模型名 + 资产信息 + 分镜信息）+ `visualManual`（视觉手册）；
5. 调 `u.Ai.Text("universalAi").invoke()` 生成提示词文本；
6. 写回 `o_videoTrack.prompt`，状态置"已完成"。

### 2.2 模板自适应规则（重点）

模板选择逻辑在 `generateVideoPrompt.ts`（约 104–144 行）和 `batchGeneratePrompt.ts`（39–79 行），优先级如下：

```
① 数据库绑定优先：o_modelPrompt 里按 vendorId + model 查到绑定文件 → 直接用
② 没绑定 → 按"模型名 + mode"自动匹配 data/modelPrompt/video/ 下的文件：
   - 模型名含 "wan" 且含 "2.6"           → wan2.6Single-imageFirstFrameMode.md
   - 模型名匹配 /seedance.*2[.\-]0/i      → seedance2Multi-parameterMode.md
   - mode 为 startEndRequired/endFrameOptional/startFrameOptional → universalFirstAndLastFrameMode.md
   - mode 为 JSON 数组字符串（多参）       → universalMulti-parameterMode.md
③ 都没匹配上 → 回退到 o_prompt 表 type=videoPromptGeneration 的通用模板（useData 优先，否则 data）
```

这四个模板文件真实存在于 `data/modelPrompt/video/`：

| 文件 | 适用 | 输出特点 |
|---|---|---|
| `wan2.6Single-imageFirstFrameMode.md` | Wan 2.6 单图首帧 | 严格按 `videoDesc` 12 字段（画面/场景/景别/运镜/动作/情绪/光影/台词/音效…）生成 |
| `universalFirstAndLastFrameMode.md` | 通用首尾帧 | 同 12 字段，但语义偏"从首帧到尾帧的过渡描述" |
| `universalMulti-parameterMode.md` | 通用多参 | 多图参考语境，资产 ID ↔ 角色标签映射 |
| `seedance2Multi-parameterMode.md` | Seedance 2.0 | 三段论（①主体定义 ②镜头分镜 ③风格+约束包），**无分镜图、资产图为唯一参考** |

> **这意味着什么**：模板是按"模型名特征 + mode"硬匹配的。如果你用的是 Wan 2.6 但项目 `mode` 配成了多参，模型名匹配优先级更高——会走 `wan2.6Single-imageFirstFrameMode.md`（单图模板），而你期待的多参语境就没了。**模型名和 mode 要自洽**，否则模板和实际能力对不上。

### 2.3 所有模板共有的硬约束（后期必须知道）

四个模板都强制了几条铁律，理解它们能帮你预判 AI 输出：

- **台词不可缺失、不可翻译**：`videoDesc` 里有台词的分镜，提示词必须原样保留台词（不翻译成英文）。
- **台词类型必须标注**：区分对白（dialogue/说）、内心独白（OS）、画外音（VO）。这对后期配音怎么处理至关重要——OS/VO 不需要对口型。
- **时间分段最低 1 秒**：任何时间分段不得低于 1s。
- **不编造、不改原始输入**：严格基于 `videoDesc` 12 字段，不自行加戏。

所以——**如果生成的视频没台词、运镜不对、加了莫名其妙的内容，根上多半是 `videoDesc` 写得不对（分镜阶段的问题），而不是提示词模板的问题。** 提示词模板只是忠实翻译。

### 2.4 手动改提示词：`updateVideoPrompt`

AI 生成的提示词不满意，可以手改。前端调 `updateVideoPrompt.ts`：

```
POST /api/production/workbench/updateVideoPrompt
{ id: <trackId>, prompt: "你改好的提示词" }
```

它只更新 `o_videoTrack.prompt` 一个字段。**注意：它不重新生成视频，只改提示词文本。** 改完要重新点生成视频才生效。

**什么时候该手改？**

- 运镜不对（AI 写了"固定镜头"，你想要"缓慢推进"）；
- 台词节奏要微调（断句、停顿提示）；
- 想强化某个细节（光影、情绪）；
- AI 误解了某个资产标签。

**Before / After 示例**：

| | 内容 |
|---|---|
| Before（AI 生成） | 男主站在办公室落地窗前，转身面对镜头，说"这家公司，我要定了"。固定镜头，3秒。 |
| After（后期手调） | 男主背对镜头站在办公室落地窗前（0-1s），缓慢转身，镜头同步轻微推近至胸像景别（1-2.5s），转身完成后定睛直视镜头，冷峻表情，说"这家公司，我要定了"（dialogue）（2.5-3s）。冷色调，逆光剪影感，3秒。 |

后者把运镜、景别变化、时间分段、台词类型都点明了，出片的"电影感"会明显更强。

### 2.5 时长 `updateVideoDuration` 对节奏的影响

时长落在 `o_videoTrack.duration`，改它走 `updateVideoDuration.ts`（只更新 `duration` 字段）。也可以在新建轨道时（`addTrack.ts`）带上 `duration`。

时长既影响**生成参数**（最终传给 `aiVideo.run({ duration })`），也影响**提示词内部的节奏把控**（模板会按 duration 拆时间分段）。

| duration | 适合 | 坑 |
|---|---|---|
| 3s | 快切镜头、反应镜头、空镜 | 太短装不下长台词，AI 会硬塞导致语速失真 |
| 5s | 标准对话/动作镜头（最常用） | — |
| 8~10s | 长镜头、情绪铺陈、复杂运镜 | 越长 AI 越容易"动作发散"或后半段静止/鬼畜 |

> **后期经验**：AI 视频模型普遍"前 5 秒稳，后面容易飘"。需要长镜头时，优先考虑拆成两条轨道（两个 5s）再剪接，而不是硬生成一条 10s。台词长度也要和 duration 配——一句 5 秒的台词配 3 秒视频，要么语速被压快、要么口型对不上。

**重要约束**：`duration` 和 `resolution` 不是随便填的。每个视频模型在 `vendor.json` 模板里有 `durationResolutionMap`（`{ duration: number[]; resolution: string[] }[]`），声明了它支持的时长/分辨率组合。填了模型不支持的组合，生成会失败。前端通常会按所选模型限制可选项，但你手动改时要留意。

### 2.6 duration / resolution / aspectRatio 取舍

这三个参数最终都进 `aiVideo.run()`（见 `generateVideo.ts` 94–104 行）：

| 参数 | 来源 | 取舍 |
|---|---|---|
| `duration` | `o_videoTrack.duration` | 见上 |
| `resolution` | 前端传入（受模型 `durationResolutionMap` 约束） | 越高越清晰越慢越贵；分镜常规即可，关键镜头再上高分辨率 |
| `aspectRatio` | `o_project.videoRatio`（默认 `16:9`） | 短剧/漫剧用 `9:16`。**这个在项目设置就该定死**，中途改要重出 |

## 三、配音与音频

### 3.1 角色音色绑定：`o_assetsRole2Audio`

这是一张多对多关系表（`src/lib/initDB.ts` 1028 行）：

```
o_assetsRole2Audio: { assetsRoleId, assetsAudioId }
```

把"角色资产"绑到"音频资产"上——也就是给每个角色指定一条嗓子。绑定方式：

- **自动绑定**：`src/routes/cornerScape/batchBindAudio.ts`。它把项目里 `type=audio` 且未挂主体的音频资产列成候选，让 AI（`universalAi`）按角色名/描述匹配最合适的音色，调 `resultTool` 写入 `o_assetsRole2Audio`，并把角色的 `audioBindState` 置"已完成"。**前提：资产中心里得先有音频资产**（`batchBindAudio.ts` 28 行：没有音频会直接报"请先前往资产中心上传音频"）。
- **手动改绑**：`src/routes/cornerScape/updateAssetsAudio.ts`。
- **查绑定**：`getAudioBindAssetsList.ts`（按角色资产 ID 查它绑了哪条音频）。

### 3.2 音频怎么进视频生成（`audioReference`）

理解这条链路是配音的关键。在 `generateVideoPrompt.ts`（86–97 行）：

1. 找出资产里 `type=audio` 的；
2. 通过 `o_assetsRole2Audio` 反查：哪个角色（`assetsRoleId`）绑了哪条音频（`assetsAudioId`）；
3. 在提示词的资产信息里，把音频挂在对应角色后面，标成 `audio:<id>`。

然后在真正生成视频时（`generateVideo.ts` 71–76 行），音频文件被转成 base64，**`type` 标成 `"audio"`**，作为 `referenceList` 的一员传给模型：

```ts
{ base64: ..., type: item.sources == "audio" ? "audio" : "image" }
```

而 `ReferenceList` 类型（`ai.ts` 230 行）支持 `image`/`audio`/`video` 三类。**但音频要真正被模型当"音色/对口型"用，前提是 `mode` 里有 `audioReference:K`**——否则就算音频进了 referenceList，模型也不会按音频驱动。这就是 1.3 节强调"`audioReference:K` 驱动放几条音频"的原因。

### 3.3 TTS 现状

底层有 TTS 能力：`u.Ai.Audio(model).run(ttsRequest)`（`ai.ts` 的 `AiAudio` 类，324–351 行），供应商模板里有 `TTSModel`（带 `voices: { title, voice }[]`），`vendor.json` 里约 12 处 `ttsRequest`。但要注意：

- **音频资产是"先有文件、再绑角色"的模式**——主流程是上传/已有音频资产 → 自动/手动绑到角色 → 作为 `audioReference` 喂给视频模型。
- `AiAudio.run` 的 `catch` 块是**静默吞异常**的（339 行 `catch (e) {}`），TTS 失败不会抛错。用到 TTS 链路时，别指望它给你明确的错误——要主动检查产物是否真的生成了。

### 3.4 对口型与节奏的注意点

- **对口型依赖模型 + mode**：只有"模型支持 `audioReference` 且项目 mode 配了它 + `audio: true`"时，才有对口型可言。不是所有视频模型都支持音频驱动口型。
- **OS/VO 不需要对口型**：模板会把台词标 dialogue/OS/VO。内心独白和画外音不需要人物开口，配音时按旁白处理。
- **台词时长 vs 视频时长**：前面说过，台词长度要和 `duration` 匹配。配音前先估一下"这句话念出来几秒"，再定 duration。
- **音色一致性**：同一角色全片要用同一条音频绑定，别这条镜头绑 A 音色、那条绑 B，否则观众会觉得"换人了"。

### 3.5 BGM / 音效现状

工作台目前**没有独立的 BGM / 音效轨**。音效信息存在于分镜的 `videoDesc`（第 11 字段"音效"），会被写进视频提示词，由视频模型在生成时"尽量体现"——但这是模型层面的，不是可控的音效轨。**真正的 BGM、转场音效、混音，目前需要导出成片后在外部剪辑软件里做。** 别指望工作台帮你铺背景音乐。

## 四、工作台协作：轨道、候选、批量

### 4.1 轨道增删

| 操作 | 路由 | 行为 |
|---|---|---|
| 新建轨道 | `addTrack.ts` | 以 `Date.now()` 为 `id` 插入 `o_videoTrack`，可带 `duration` |
| 删除轨道 | `deleteTrack.ts` | 删 `o_videoTrack`，并把挂在它下面的 `o_storyboard.trackId` 置 null（解绑分镜） |

> 删轨道会把分镜从这条轨道解绑（`trackId=null`），但不删分镜本身。分镜还在，只是不再属于任何轨道。

### 4.2 一镜多候选 → 选片

这是轨道制的精髓：

- 同一条 track 可以反复生成视频，每次生成都往 `o_video` 插一条新记录（`videoTrackId` 指向这条 track）；
- 这些候选在工作台同一镜头下并排展示（`getGenerateData.ts` 把 `videoList` 按 `videoTrackId` 分组挂到每条 track）；
- 满意哪条，调 `selectVideo.ts` 把 `o_videoTrack.videoId` 设为该候选 ID——这条就是成片里用的；
- 不要的候选用 `delVideo.ts` 删（删 `o_video`，并把任何引用它的 `videoTrack.videoId` 置 null）。

**后期工作流建议**：关键镜头（开场、高潮、人物特写）一次生成 2~3 个候选挑最好的；过场/空镜生成 1 个够用。AI 视频有随机性，重要的镜头多抽几次卡牌。

### 4.3 批量生成与并发

| 批量操作 | 路由 | 并发控制 |
|---|---|---|
| 批量生成提示词 | `batchGeneratePrompt.ts` | `pLimit(concurrentCount ?? 5)`，默认并发 5，后台跑不阻塞响应 |
| 批量生成视频 | `batchGenerateVideo.ts` | **无 pLimit**，`for` 循环里全部 `aiVideo.run().then()` 并发后台执行 |

注意这两者的并发模型不同：

- **提示词批量**用了 `pLimit`，并发数可控（`concurrentCount`），相对温和；
- **视频批量**是"全部一次性并发"——`batchGenerateVideo.ts` 先同步把所有 track 的 `o_video` 记录插成"生成中"并立即返回，然后在 `for` 循环里逐个发起 `aiVideo.run()` 但**不 await**，等于把所有视频任务同时丢给供应商。

> **坑提醒**：视频批量没有并发上限。如果你一次性批量生成几十条镜头，会瞬间向供应商发起几十个视频任务——可能触发供应商的并发/速率限制，导致一批任务集体失败。**建议分批批量**（一次十几条），或在前端控制每批数量。

### 4.4 状态查询与失败重试

每条视频/轨道都有状态字段：

- `o_video.state`：`生成中` / `生成成功` / `生成失败`（失败原因在 `errorReason`）；
- `o_videoTrack.state`：`未生成` / `生成中` / `已完成` / `生成失败`（提示词失败原因在 `reason`）。

查询路由：

| 路由 | 查什么 |
|---|---|
| `checkVideoStateList.ts` | 按 `videoIds` 查视频生成状态（只返回"生成成功/失败"的，带可访问 `src`） |
| `checkVideoPrompt.ts` | 按 `trackIds` 查提示词状态（"已完成/生成失败"，带 `prompt`/`reason`） |
| `getVideoList.ts` | 按分镜列表查所有相关视频（缩略图 url） |
| `getGenerateData.ts` | 工作台主数据（分镜 + 轨道 + 每条轨道的候选视频列表 + 资产媒体） |

**失败重试怎么做**：

- 视频生成失败 → 看 `o_video.errorReason`，对症（改提示词/换 mode/降时长/检查参考图）后，对同一 track 重新生成（产生新候选），删掉失败的旧候选。
- 提示词生成失败 → 看 `o_videoTrack.reason`，重跑 `generateVideoPrompt`。
- **没有"自动重试"机制**——失败的任务就停在"生成失败"状态，必须人工触发重生成。批量任务里失败的那几条不会自动重跑。

## 五、成片合成与导出现状

### 5.1 成片是怎么"连"起来的

成片的顺序由**分镜顺序**决定。`getVideoList.ts` 和 `getGenerateData.ts` 都按 `o_storyboard.index` 升序排分镜，每个分镜对应一条 track，track 的选定视频（`videoId`）就是这一段成片。也就是说：

```
分镜 index 顺序 → 轨道顺序 → 每轨选定的 o_video → 顺次拼接 = 成片
```

### 5.2 当前能力与边界（务必知道）

| 能力 | 现状 |
|---|---|
| 单镜头视频生成 | ✅ 完整 |
| 一镜多候选 + 选片 | ✅ 完整（`selectVideo`） |
| 按分镜顺序组织 | ✅（index 排序） |
| 角色音色绑定 + audioReference | ✅（需模型支持） |
| BGM / 音效轨 | ❌ 无独立轨（音效仅靠 prompt 体现） |
| 字幕 | ❌ 工作台无字幕功能；台词在 prompt 里，烧字幕需外部处理 |
| 转场效果 | ❌ 无（硬切，转场需外部剪辑） |
| 时间线精剪 / 变速 / 裁切 | ❌ 无；工作台是"镜头级"而非"帧级"编辑 |
| 一键导出整片 | 以"逐镜头视频 + 选定关系"为产物，最终合成/导出依赖外部流程 |

### 5.3 实操建议

1. **把工作台当"镜头工厂"，把剪辑软件当"成片台"**。工作台负责把每个镜头生成好、选好；字幕、BGM、音效、转场、精剪到剪映/PR/达芬奇里做。
2. **导出前先在工作台把每条 track 的 `videoId` 都选定**，确认没有遗漏的"未选"镜头。
3. **统一画幅在项目层定死**（`videoRatio`），不要混用，否则拼接时尺寸打架。
4. **台词字幕从分镜 `videoDesc` 的台词字段提取**（带 dialogue/OS/VO 标注），这是现成的字幕底稿，导出后照着烧。

## 本章排障表（症状 → 病因 → 改哪里）

| 症状 | 可能病因 | 改哪里 |
|---|---|---|
| 视频几乎不动 / 太静 | 提示词没写运镜和动作；duration 太长后半段静止 | 手改 `updateVideoPrompt` 加运镜/动作分段；或缩短 `duration` |
| 参考图没生效、人物跑形 | mode 选错（如该用 `imageReference:N` 却用了 `text`/`singleImage`）；图没传进去 | 检查 `o_project.mode` 与模型能力；确认 `uploadData` 传了资产 |
| 多给的参考图被忽略 | `singleImage` 只用第一张；或超过 `imageReference:N` 的 N 被截断 | 改用多参 mode 并把 N 调够 |
| 台词没出现在视频里 | 分镜 `videoDesc` 没填台词字段；或被手改提示词时删了 | 回分镜补台词；检查 track.prompt |
| 口型对不上 | 模型不支持音频驱动；mode 没配 `audioReference:K`；`audio` 没开 | 换支持的模型；mode 加 `audioReference:K`；生成时 `audio: true` |
| 没声音 / 音色不对 | 角色没绑音频（`o_assetsRole2Audio` 空）；绑错音色 | `batchBindAudio` 或 `updateAssetsAudio` 重新绑 |
| 时长不对 / 生成报时长错误 | duration/resolution 组合不在模型 `durationResolutionMap` 内 | 改成模型支持的组合 |
| 提示词模板不对（多参却出单图语境） | 模型名特征优先匹配（如 wan2.6 强制走单图模板） | 让 `videoModel` 与 `mode` 自洽；或在 `o_modelPrompt` 绑定指定模板 |
| 批量生成大面积失败 | 视频批量无并发上限，撞供应商速率限制 | 分批生成（每批十几条） |
| 失败的镜头没自动重来 | 无自动重试机制 | 看 `errorReason`/`reason`，人工对同一 track 重新生成 |
| 成片有黑边 / 比例乱 | `videoRatio` 中途改过或混用 | 项目层统一画幅，重出不一致的镜头 |
| 想加 BGM/字幕/转场 | 工作台不支持 | 导出后外部剪辑软件处理 |
