# 第09章 会话与记忆管理：和 AI 高效协作的"导演沟通术"

## 本章导览

你在 Chat 里跟剧本 Agent / 制作 Agent 来回对话，本质上是在**和一个有记忆的合作者沟通**。这个合作者记什么、忘什么、怎么记，直接决定了它产出稳不稳、跑不跑题。

很多创作者把对话框当一次性的"提需求"窗口，聊崩了就猛点"清空"，结果越清越乱——因为他们不知道那几个按钮各自动了什么。

这一章把 Toonflow 的记忆系统讲透，基于真实代码：`src/utils/agent/memory.ts`（三层记忆引擎）和 `src/routes/agents/clearMemory.ts`（清理逻辑）。读完你会知道：

- AI 到底"记得"什么、怎么记的（用创作者能懂的话）；
- "重新连接 / 清空消息记忆 / 清空摘要记忆 / 清空全部记忆"四个操作的**精确行为**、适用场景、决策树、误区；
- 记忆的**隔离范围**（每项目每 Agent 独立）对你的协作意味着什么；
- 怎么用"对话节奏"配合记忆系统，拿到更稳的产出；
- 那几个藏在设置里的可调参数（`messagesPerSummary` 等）调了会怎样。

---

## 一、三层记忆：AI 的"工作笔记本"长什么样

打开 `src/utils/agent/memory.ts`，AI 的记忆其实是三层叠在一起的（`get()` 方法一次性把三层都取出来拼进上下文，见各 Agent 的 `buildMemPrompt`）。用拍片的比喻来理解：

| 层 | 代码里的名字 | 通俗比喻 | 取多少条（默认） |
|---|---|---|---|
| 短期记忆 | `shortTerm` | 导演手边刚说过的几句话，逐字记得 | 最近 **未被摘要** 的 message，约 **5** 条 |
| 摘要记忆 | `summaries` | 场记把更早的对话压成的"会议纪要" | 最近 **10** 条摘要 |
| RAG 语义检索 | `rag` | 翻遍全部历史，捞出"和你这句话最相关"的几条 | 与当前输入最相似的 **3** 条 message |

它们是怎么运转的：

1. **你每说一句、AI 每答一段，都会作为一条 `message` 存进记忆**（`add()`）。
2. 当**未摘要的 message 累积到阈值**（`messagesPerSummary`），系统自动把最早的一批压成一条**摘要**（不超过 `summaryMaxLength` 字），并把这批 message 标记为"已摘要"。
   - 一旦被摘要，它们就**退出短期记忆**，只以"纪要"的形式存在于摘要层。
3. 下次对话时，AI 看到的"记忆"= 最相关的几条原话（RAG）+ 若干条纪要（摘要）+ 最近几句原话（短期）。

> 关键直觉：**短期 = 逐字、近、少；摘要 = 概括、远、多；RAG = 不论远近、只论相关。** 三者互补，让 AI 既记得刚说的细节，又不至于忘了三十轮前定下的设定。

还有一个进阶机制：`deepRetrieve`（深度检索）。它是 AI **主动**用的工具——当它需要回忆某个关键词的历史细节时，会先在摘要里语义检索，再让模型判断哪些纪要相关，最后**展开还原成当时的原始对话**。你不用管它，知道"AI 有能力主动翻旧账找细节"即可。

## 二、四个按钮，到底各动了什么

这一节是本章的核心。先看三个"清空"按钮的**精确行为**（全部来自 `clearMemory.ts`，按 `type` 分支）：

```ts
isolationKey = `${projectId}:${agentType}${episodesId ? `:${episodesId}` : ""}`
```

### 2.1 清空全部记忆（`type = "all"`）
```ts
await u.db("memories").where({ isolationKey }).del();
```
**行为**：把这个隔离范围下的**所有记忆全删**——短期、摘要、全没了。AI 彻底失忆，下一句话开始像第一次见面。

### 2.2 清空消息记忆（`type = "message"`）
```ts
await u.db("memories").where({ isolationKey, type: "message" }).del();
await u.db("memories").where({ isolationKey, type: "summary" }).del();
```
**行为**：删掉所有 message，**并连带删掉所有 summary**。代码注释说得很清楚——*"删 message 时同步删关联的 summary，避免悬挂引用"*。

> ⚠️ **重要误区**：很多人以为"清空消息记忆"只清原话、保留纪要。**不是**。它会把摘要一起带走。从结果看，它和"清空全部"非常接近（都让 AI 几乎归零）。区别仅在实现细节，对你而言：**想保留纪要就别点它。**

### 2.3 清空摘要记忆（`type = "summary"`）
```ts
await u.db("memories")
  .where({ isolationKey, type: "message", summarized: 1 })
  .update({ summarized: 0 });          // 被压缩的 message 重新变"未摘要"
await u.db("memories").where({ isolationKey, type: "summary" }).del();  // 删摘要
```
**行为**：这是最"温柔"也最聪明的一个。它删掉所有摘要，**同时把那些曾被摘要压缩掉的原始 message 重置为"未摘要"**——也就是说，**那些原话被"放回"了短期记忆的候选池**。

效果：AI 不再依赖那些可能"压糊了"的纪要，而是回到逐字的原话。代价是上下文会变长（原话比纪要占字数多），到了阈值它会**重新压缩一次**（用当前更新过的设定/模型重做摘要）。

> 通俗讲：**"清空摘要记忆" = 把走样的会议纪要撕了，让 AI 重新读原始录音。** 信息没丢，只是换了一种保存形态。

### 2.4 重新连接（不动记忆）
**行为**：只**重建 socket 连接 + 中断当前正在进行的生成**。它**完全不碰记忆**——一条都不删。

适用场景：生成卡住了、流断了、AI 跑飞了想刹车。点它 = "咔，停，重来一条"，但剧本设定、之前聊的一切都还在。

### 2.5 四者对比

| 操作 | 短期 message | 摘要 summary | AI 还记得之前的事吗 | 典型用途 |
|---|---|---|---|---|
| 重新连接 | 不动 | 不动 | **全记得** | 中断卡死/跑飞的生成，刷新连接 |
| 清空摘要记忆 | 保留（被摘要的原话放回候选池，待重压缩） | 删除 | **基本记得**（回到原话） | 摘要把设定压走样了，想让 AI 重读原话 |
| 清空消息记忆 | 删除 | **连带删除** | 几乎不记得 | 想重开但又点错了（多数情况该用"全部"） |
| 清空全部记忆 | 删除 | 删除 | **完全失忆** | 彻底换故事/换思路，干净重来 |

## 三、什么时候点哪个：决策树

```
AI 这次产出不对劲？
│
├─ 是"卡住/跑飞/流断了"，设定本身没问题？
│     → 重新连接（保留记忆，止损重来一条）
│
├─ 是"它把我早先定的设定记串了/压糊了"，但大方向还想接着聊？
│     → 清空摘要记忆（撕掉走样纪要，回到原话，信息不丢）
│
├─ 是"这条故事线我想推倒重来"，之前的全不要了？
│     → 清空全部记忆（彻底归零）
│
└─ 想"保留纪要只删原话"？
      → 没有这个选项。"清空消息记忆"会连摘要一起删，
        想干净重开就直接用"清空全部记忆"，别绕。
```

**头号误区，必须单独拎出来强调：**

> **清记忆 ≠ 删产物。** 清空记忆只清空 Chat 里 AI 的"对话记忆"，**不会删除你已经生成的剧本、骨架、资产、分镜、视频**——那些是落库的产物（`o_script`、`o_storyboard`、`o_assets` 等），和 `memories` 表完全是两码事。放心大胆地清记忆，你的成片资产动都不会动。

第二个常见误区：**遇事就清全部**。记忆是 AI 理解你的资本。动不动清空，等于每次都让一个失忆的新人从头接手项目，产出当然忽好忽坏。**清记忆是手术刀，不是橡皮擦。**

## 四、隔离范围：每项目、每 Agent、（可选）每集 各记各的

看 `isolationKey` 的构成：`{projectId}:{agentType}[:{episodesId}]`。这意味着记忆是**严格隔离**的：

- **不同项目**之间，记忆互不相通。项目 A 聊的设定，项目 B 的 AI 完全不知道。
- **同一项目里，剧本 Agent 和制作 Agent 各记各的**（`scriptAgent` vs `productionAgent`）。你跟编剧聊的人物小传，制作 Agent 那边**看不到**。
- 带 `episodesId` 时，还能进一步按"集"隔离。

对协作的三条实战含义：

1. **跨 Agent 的信息不会自动传递。** 你在剧本 Agent 里反复强调"男主不能说脏话"，到了制作 Agent 它并不知道——重要约束要在对应 Agent 里**各说一遍**。（项目级的设定靠的是第01章那些字段，不是聊天记忆。）
2. **清记忆是"定点清除"。** 你清空"项目X的制作Agent"记忆，不会波及同项目的剧本 Agent，更不会波及别的项目。放心清。
3. **换项目=换了个全新合作者。** 别指望新项目的 AI 记得你上个项目的偏好。

## 五、用"对话节奏"配合记忆系统，拿更稳的产出

懂了机制，就能反过来"喂"得更聪明：

1. **重要设定尽早、成块地说清楚。** 越早说，越可能在后续被压成稳定的"纪要"沉淀下来；零碎地分二十句挤牙膏，容易在摘要时被裁掉细节。
2. **一句话别塞太多任务。** 短期记忆只逐字保留最近约 5 条，信息过载会把更早的有用原话挤出短期层（进入摘要被压缩）。一次聊一件事，节奏清楚。
3. **关键约束定期"复述"一次。** 因为 RAG 是按"相关性"召回的，你在关键节点重申"记住，结局是 BE"，能提高它在后续被召回的概率。
4. **发现 AI 开始"记串"了，先想清空摘要、而不是清空全部。** 走样往往出在摘要层（压缩损失），撕掉重压通常就好，没必要全失忆。
5. **生成跑飞别硬聊，先重新连接止损。** 在一条已经歪掉的生成上继续追问，只会把错误也写进记忆。

## 六、藏在设置里的记忆参数（进阶）

`memory.ts` 顶部定义了一组可调项，运行时通过 `getConfigData()` 从 **`o_setting` 表**读取（数据库里有值就用数据库的，否则用代码默认值）。这些 key 在 `o_setting` 里都有种子值（见 `src/lib/initDB.ts`）：

| 参数（`o_setting` 的 key） | 控制什么 | 代码默认 | `o_setting` 种子值 | 调大的影响 | 调小的影响 |
|---|---|---|---|---|---|
| `messagesPerSummary` | 攒多少条 message 触发一次摘要 | 3 | **10** | 摘要更少、原话保留更久（更逐字、上下文更长） | 频繁压缩，更早进入"纪要化"，更省上下文但更早丢细节 |
| `summaryMaxLength` | 单条摘要最大字符数 | 500 | 500 | 摘要更详细、信息保留多，但更占上下文 | 摘要更精炼，可能丢细节 |
| `shortTermLimit` | `get()` 返回的近期未摘要 message 条数 | 5 | 5 | AI 逐字记得更多近期对话 | 逐字记得更少，更依赖摘要 |
| `summaryLimit` | `get()` 返回的摘要条数 | 10 | 10 | 带入更多历史纪要 | 历史视野变窄 |
| `ragLimit` | 向量检索返回的相关 message 条数 | 3 | 3 | 召回更多相关旧对话 | 召回更少，更聚焦近期 |
| `deepRetrieveSummaryLimit` | 深度检索时召回的摘要条数 | 5 | 5 | 深挖时覆盖更广 | 深挖更聚焦 |

> 注意：**数据库种子里 `messagesPerSummary` 是 10，而代码默认是 3**——以数据库为准，所以你的实际系统多半是"攒 10 条才摘要一次"。这是个有意的取舍：保留更长的逐字上下文。

调参心法（给愿意折腾的人）：
- **嫌 AI 忘细节太快** → 调大 `messagesPerSummary`、`shortTermLimit`（让原话活得更久）。代价是上下文变长、成本变高、超长可能触发模型限制。
- **嫌 AI 上下文太臃肿/成本高** → 调小这些值，让它更早压缩、更精简。代价是更容易丢细节。
- **嫌 AI"翻旧账"翻不准** → 调大 `ragLimit` / `deepRetrieveSummaryLimit`。
- `summaryMaxLength` 是质量与成本的旋钮：太小摘要会糊，太大失去"压缩"的意义。500 是个合理起点。

改完一般无需重启逻辑（每次 `get()`/`add()` 都现读 `o_setting`），但**已经生成的旧摘要不会自动按新参数重做**——想让新参数对历史生效，可以对相应隔离范围"清空摘要记忆"，触发用新参数重新压缩。

## 本章调优速查表

| 场景 | 该用的操作 | 为什么 |
|---|---|---|
| 生成卡死/跑飞/流断 | 重新连接 | 止损重来，记忆全保留 |
| AI 把设定记串了，但想接着聊 | 清空摘要记忆 | 撕掉走样纪要、回到原话，信息不丢 |
| 整条故事线推倒重来 | 清空全部记忆 | 彻底归零 |
| 想"只删原话保留纪要" | 没有此选项 | "清空消息记忆"会连摘要一起删 |

| 认知要点 | 一句话 |
|---|---|
| 清记忆会删产物吗？ | **不会**。剧本/分镜/资产/视频都在各自的表里，安然无恙 |
| 记忆怎么隔离？ | 每项目、每 Agent（可到每集）独立，互不相通 |
| 剧本 Agent 的话制作 Agent 知道吗？ | 不知道，重要约束要各说一遍 |
| 三层记忆 | 短期（逐字近少）+ 摘要（概括远多）+ RAG（按相关召回） |
| 摘要触发阈值 | `messagesPerSummary`（系统种子值=10） |
| 调参在哪 | `o_setting` 表，运行时现读，改完对历史摘要可"清空摘要记忆"令其重做 |

最后一句制片人的叮嘱：**把 AI 当成一个会记笔记的搭档，而不是一次性的搜索框。** 设定说清楚、节奏带稳、出问题先用对的那把"手术刀"——你会发现它的产出比你猛点"清空全部"时稳定得多。
