This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

Toonflow is an AI short video creation tool that converts novels into scripts and then into storyboards, images, and videos using AI models. It runs both as an **Electron desktop app** and as a **Node.js/Docker web server**. The frontend is a separately-built static Vue/React app served from `data/web/`.

## Commands

```bash
# Development
yarn dev              # Run as Node.js server with hot reload (nodemon + tsx), port 10588
yarn dev:gui          # Run as Electron desktop app with hot reload

# Type checking (no test suite exists)
yarn lint             # tsc --noEmit (TypeScript type check only)

# Build
yarn build            # Bundle via esbuild → data/serve/app.js + build/main.js
yarn dist             # yarn build + electron-builder package
yarn dist:mac         # macOS distribution
yarn dist:linux       # Linux distribution

# Utilities
yarn debug:ai         # Launch @ai-sdk/devtools inspector
```

No test runner is configured. `yarn lint` is the only automated check.

## Architecture

### Dual-Mode Runtime

The app boots from `src/app.ts`. At runtime it detects whether it's running under Electron or as a plain Node process, then adapts data paths and behavior accordingly. `u.getPath(subdir?)` centralizes all path resolution—always use it instead of hardcoding paths.

### Auto-Generated Router

**Do not edit `src/router.ts` manually.** It is generated by `src/core.ts`, which scans `src/routes/**/*.ts` and writes the router file on startup when `NODE_ENV=dev`. File path → API route mapping is direct: `src/routes/novel/addNovel.ts` → `/api/novel/addNovel`. Adding a new route file is enough; the router auto-registers it.

### Database

SQLite via **Knex** + **better-sqlite3**. Database lives at `data/db2.sqlite`.

- Schema and seed data are in `src/lib/initDB.ts`.
- Migrations/fixes go in `src/lib/fixDB.ts`.
- In dev mode, TypeScript types for all tables are auto-generated at `src/types/database.d.ts` from live schema—do not edit that file by hand.
- Access the DB via `u.db("table_name")` (typed using the generated `DB` interface).

### AI Agent System

Two main agents, each with a layered architecture:

- **Script Agent** (`src/agents/scriptAgent/`): Decision layer → supervision layer → execution sub-agents (story skeleton, adaptation strategy, script writing)
- **Production Agent** (`src/agents/productionAgent/`): Decision layer → supervision layer → execution sub-agents (storyboard, assets, video prompts)

Agents communicate with the frontend over **Socket.IO** using `ResTool` (`src/socket/resTool.ts`) to stream text, thinking blocks, and tool results in real time.

**Skills** are markdown prompt files stored in `data/skills/`. Agents read their system prompts from these files at runtime—editing a skill file takes effect immediately without rebuilding.

### Memory System

Agents use a three-tier memory system (`src/utils/agent/memory.ts`):

1. **RAG** – semantic search over past conversations using local `all-MiniLM-L6-v2` ONNX embeddings
2. **Summaries** – LLM-generated rolling summaries of older conversations  
3. **Short-term** – recent messages verbatim

### AI Model Providers

Multiple providers via the **Vercel AI SDK** (`ai` package): OpenAI, Anthropic, DeepSeek, Google, xAI, Minimax, Zhipu, Qwen, and custom compatible endpoints. Provider credentials and model selection are stored in the DB (`o_vendorConfig`, `o_agentDeploy`). `u.Ai.Text(agentKey)` resolves the configured model and returns a ready-to-use AI client.

### OSS / Static Assets

Generated images and files are stored under `data/oss/`. The `/oss` endpoint supports on-demand thumbnail generation via `?size=WxH` or `?size=percentage` query params, using **sharp** under the hood.

## Docker / Deployment

The Dockerfile is multi-stage: builder compiles `data/serve/app.js`, runner copies only production artifacts. The `data/` directory is a **Coolify persistent volume**—the entrypoint script seeds `data/web/` and `data/models/` from the image on first boot if they don't exist. Do not include Electron-related packages in the Docker build (they're stripped from `package.json` during the builder stage).

## Development Notes

- `@/*` path alias maps to `src/*` (configured in both `tsconfig.json` and esbuild).
- Native modules (`better-sqlite3`, `sharp`, `@huggingface/transformers`) are marked external in esbuild and must be available in `node_modules` at runtime.
- The default login is `admin` / `admin123` (seeded in `o_user`); the JWT secret (`tokenKey`) is auto-generated on first DB init.
- When adding new DB columns, put the `ALTER TABLE` logic in `src/lib/fixDB.ts`—`initDB` only runs for new tables/columns that don't exist yet.
